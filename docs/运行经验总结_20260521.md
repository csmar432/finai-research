## 二、发现的问题

### 2.1 脚本执行问题

#### 问题1：`paper_tariff_research.py` 挂起
**现象**：运行后超过5分钟无输出，最终被后台处理。
**原因分析**：
- 脚本导入了 `from scripts.ai_router import AI, Task`，但 AI router 在初始化时需要访问网络
- Python 模块的延迟导入顺序问题——`from` 语句在模块顶层执行时会立即触发初始化
- 在 Cursor 的 Shell 环境中，标准输入输出可能被重定向，导致 `input()` 或类似阻塞调用行为异常

**解决方案**：
1. 直接使用 OpenAI SDK 而非通过 AI Router 包装层调用
2. 使用 `python-dotenv` 而非手动解析 `.env.local`
3. 提前测试 API 连接，确认模型名可用

#### 问题2：API 模型名称错误
**现象**：`deepseek-v4-pro` 返回空内容。
**原因**：配置文件中 `model: "deepseek-v4-pro"`，但实际 API 支持的模型名是 `deepseek-chat`。
**解决方案**：先测试不同模型名，确认可用后再调用。

#### 问题3：`review_paper.py` 中 DeepSeek 审查打分偏低
**现象**：审查评分 4.2/10，指责数据为虚构。
**原因**：这是 AI 审查的固有局限性——它无法真正验证数据来源，只能从形式上判断。模拟数据被当作"虚构数据"扣分。
**改进**：对于已知基于模拟/示例数据的论文写作任务，应跳过或调整审查标准。

### 2.2 论文格式问题

#### 问题4：Markdown 表格不是标准学术表格
**现象**：原始 Markdown 使用 `|` 分隔符表格，Word 中自动转换后无边框线。
**原因**：`python-docx` 的 Markdown 导入功能不完整，表格被解析为空段落。
**解决方案**：完全抛弃自动转换，编写专门的 `md_to_docx` 函数，逐行解析并手动构建 Word 表格对象。

#### 问题5：表格格式不统一
**现象**：不同表格列数、行结构不一致（如有的带粗体行标题）。
**改进**：制定统一的表格规范（三线表），强制所有表格遵循同一格式。

### 2.3 AI写作质量问题

#### 问题6：论文存在明显 AI 写作特征
**现象**：
- "本文的研究发现表明"、"这一发现深刻揭示了"等套话
- 贡献部分过度使用"第一、第二、第三"
- 句式单一、重复开头词汇多
- "然而"、"但是"使用频率过高

**解决方案**：
1. 生成后使用专门的去AI痕迹 prompt
2. 人工检查并调整句式
3. 未来可训练专属风格的 prompt 模板

#### 问题7：实证数据引用不一致
**现象**：审查阶段发现描述性统计表格中数据为模拟值，与真实实证结果描述不一致。
**根本原因**：论文内容生成与实证分析是分离的——实证结果以 CSV 表格形式存在，但论文生成时仅引用了摘要性结论，细节数据（如标准误、样本量）存在偏差。

**改进方案**：
- 将实证分析结果直接嵌入 prompt，确保论文引用数字与实证表格完全一致
- 建立"数据-论文"同步机制

## 三、工作流改进建议

### 3.1 论文写作流程（改进版）

```
输入：
├── 研究方案（knowledge/outlines/xxx.md）  ← 背景/理论/方法
├── 实证结果（tariff_research/results/tables/*.csv）  ← 数据支撑
├── 文献综述（knowledge/reviews/xxx.md）  ← 文献引用
└── 写作规范（模板/格式要求）

Step 1：生成章节（DeepSeek）
  → 输出：各章节 .md 文件

Step 2：数据一致性校验
  → 交叉比对论文中数字与实证表格
  → 修正不一致之处

Step 3：去AI痕迹（DeepSeek）
  → 输出：自然化文本

Step 4：格式转换（python-docx）
  → 输出：符合学术格式的 Word 文档
  └── 三线表（顶线/底线粗，表头线细，白底黑字）
  └── 公式（居中斜体）
  └── 参考文献（APA格式）

Step 5：终审（人工）
  → 核对数字、逻辑、格式
```

## 四、本次运行的关键经验

### 4.1 API调用策略
| 策略 | 说明 |
|------|------|
| **提前测试模型名** | 不同 API 端点支持的模型名不同，`deepseek-chat` vs `deepseek-v4-pro` |
| **直接 SDK 调用** | 避免中间层挂起问题，直接用 `openai.OpenAI()` |
| **timeout 设置** | 长文本生成需要 180-300s timeout |
| **分步调用** | 章节生成→整合→润色分开，避免单次 token 溢出 |

### 4.2 论文质量控制
| 检查点 | 方法 |
|--------|------|
| 数据一致性 | 论文数字 vs 实证CSV，交叉核对 |
| AI痕迹 | 使用去AI prompt + 人工检查 |
| 格式规范 | 三线表（python-docx 手动构建） |
| 引用格式 | 预定义参考文献样式，自动格式化 |

### 4.3 时间成本
| 操作 | 耗时 |
|------|------|
| 单次 DeepSeek API 调用（约 3000 tokens 输出） | 30-90秒 |
| 全流程（生成+润色+去AI+格式转换） | 约10-15分钟 |
| 人工终审+修改 | 取决于修改幅度 |
